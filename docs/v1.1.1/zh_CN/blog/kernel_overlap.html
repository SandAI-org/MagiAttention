如何确保 Kernel 真正实现 Overlap#

挑战#

虽然 CPU 调度器控制 kernel 的启动顺序以利于 overlap，但 GPU 的 Hyper-Q 驱动 [Bradley, 2013] 最终决定了实际的执行顺序。这一过程本质上是非确定性的，且很大程度上受 GPU 瞬时资源占用情况的影响。

因此，确保计算与通信 kernel 之间的可靠 overlap 并非易事，这主要源于 CUDA 编程中一个众所周知的难题——SM Starvation（SM 饥饿）。

计算 kernel——例如反向传播中大规模的 bwd_partial_attn 操作——天生就是“贪婪的”。当它们被调度时，可以瞬间占满 GPU 上所有可用的 Streaming Multiprocessors (SMs)。即使通信 kernel（例如网络传输）紧随其后被调度到并行的 CUDA stream 上，也找不到空闲的 SM 来执行。通信 kernel 只能在队列中持续等待，直到繁重的计算完成。

因此，在 CPU 上异步的操作在 GPU 硬件上退化为严格的串行执行——先计算，后通信——彻底打破了 overlap 的幻想。当通信 kernel 使用本身就限制并发性的 SM90+ cluster 特性时，这一饥饿问题会进一步加剧。

方案#

Single Max Connection#

先前的框架，例如 Tensor Parallelism (TP)，试图通过设置环境变量 CUDA_DEVICE_MAX_CONNECTIONS=1 [User, 2023] 来强制 GPU kernel 的严格 FIFO 调度。这保证了 GPU 驱动按照 CPU 启动的确切顺序选取通信 kernel，防止它们被长时间运行的计算 kernel 阻塞。然而，这种方法严重限制了独立 GPU stream 间的并发性，最终降低了端到端的吞吐量。因此，通常不推荐使用此方法。

SM Margin 预留#

一种专门针对 persistent compute kernel（如 FFA）的常见策略是显式预留一部分 Streaming Multiprocessors (SMs)，即 sm_margin。这种预留为通信 kernel 留出了足够的空间，使其能够与正在进行的计算并发执行。然而，配置 sm_margin 需要微妙的权衡：设置过高会牺牲计算吞吐量，设置过低则可能无法实现有效的 overlap。

根据经验，对于 AlltoAll-v-based group collectives，当 NCCL_CGA_CLUSTER_SIZE={0,1} 时，我们观察到仅将 sm_margin 设置为 4~8 即可实现完全 overlap，这比 NCCL kernel 使用的 SM 数量更少。相比之下，当 NCCL_CGA_CLUSTER_SIZE>1 或使用利用 SM90+ cluster 特性和 cooperative launch 的 native implementation 时，通信 kernel 如果未被优先调度，则需要更大的 sm_margin 才能实现 overlap——不少于其使用的 SM 数量。

备注

对于 FFA kernel，有两种方法设置 sm_margin：

  1. 如果使用 flex_flash_attn_func 接口，可以直接传递可选参数 sm_margin，该参数会被转发给底层 FFA kernel 的前向和反向传播。

  2. 如果使用 calc_attn 接口进行分布式 attention，可以设置环境变量 MAGI_ATTENTION_FFA_FORWARD_SM_MARGIN 和 MAGI_ATTENTION_FFA_BACKWARD_SM_MARGIN 来分别指定底层前向和反向 kernel 的 sm_margin。

高优先级 Stream#

对于 non-persistent compute kernel，例如 FFA_FA4，另一种直接的方法是将通信 kernel 分配到高优先级 CUDA stream。这可以促使 GPU 调度器在计算 kernel 之前调度它们，甚至可能在 wave quantization 阶段抢占正在运行的计算 kernel [NVIDIA, 2023]。然而，此方法的效果因 GPU 架构而异。例如，我们观察到它在 Hopper 架构上可靠性较低，但在 Blackwell 上效果要好得多（这值得在未来工作中进一步研究）。

备注

对于使用 PyTorch 接口的 NCCL 通信 kernel，可以简单地设置环境变量 TORCH_NCCL_HIGH_PRIORITY=1 将其分配到高优先级 stream。

Kernel Barrier#

为了彻底克服上述 SM starvation 问题，MagiAttention 引入了一个轻量级的、设备端细粒度同步原语：KernelBarrier。

与传统的 cudaEvent 机制基于 kernel 完成 进行同步（“等待通信完成”）不同，KernelBarrier 利用细粒度锁基于 kernel 启动 进行同步（“等待通信开始”）。这一微妙但强大的语义转变确保了通信 kernel 在繁重的计算“巨兽”释放之前安全地获取所需的 SM。

1. Lifecycle and Memory Management#

KernelBarrier 通过利用 PyTorch 的 RAII 机制在 host 端进行优雅管理。它在 CUDA 内存中分配一个 Int32 标量 tensor 作为计数器。这种设计确保了与 PyTorch Tensor 生命周期绑定的自动内存回收，完全消除了手动释放内存的需要（kernel_barrier.cu）。

2. The Arrive Signal in Communication Kernels#

当通信 kernel（例如用于获取远程 KV 和 QO 的 native group_cast）被调度时，barrier 的 POD 视图 KernelBarrierView 会被传入。在通信 kernel 执行的最开始——严格限定在第一个 block 的第一个 thread——它触发 arrive() 函数（kernel_barrier.cuh）：

// Executed at the very top of the communication kernel
if constexpr (kHasKernelBarrier) {
  kernel_barrier_view.arrive();
}

这个原子递增操作作为一个清晰的硬件级信号：“我已成功获取 SM 资源并开始运行。”

3. The Wait Spin-lock in the Compute Stream#

在将大规模计算 kernel 排入 Compute Stream 之前，Python host 代码通过调用 kernel_barrier_fetch.synchronize()（dist_attn.py）显式地强制同步。

这并非阻塞 CPU host，而是向 compute stream 注入一个微型 wait_kernel（仅包含 1 个 Block 和 1 个 Thread）。这个微型 kernel 执行一个 volatile while 循环（自旋锁），耐心等待直到目标数量的通信 kernel 发出到达信号。

完美 Overlap 执行流程#

这一组合在 GPU 上编排了一场完美的调度之舞：

  1. wait_kernel 被调度到 Compute Stream 上，仅占用单个 SM 的一小部分。贪婪的计算 kernel 紧随其后在队列中空闲等待。

  2. 由于 Compute Stream 被有意地暂停而不占用硬件资源，GPU 调度器可以轻松地将通信 kernel 从 Communication Stream 调度到其余空闲的 SM 上。

  3. 通信 kernel 锁定所需的 SM，开始执行，并立即触发 arrive()，递增共享计数器。

  4. 一旦达到目标到达计数（例如，远程 KV 和 QO 获取 kernel 都已启动），wait_kernel 就会跳出自旋锁并退出。

  5. 繁重的计算 kernel 立即从队列中释放，瞬间占满并独占所有剩余的空闲 SM。

通过应用这种细粒度的设备端调度技巧，MagiAttention 在硬件上安全且确定性地 overlap 了两个 stream。这有效地消除了在深度学习引擎优化中经常出现的 SM starvation 死锁问题。
